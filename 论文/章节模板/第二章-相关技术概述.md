## 2.1 分布式/系统架构基础 (约600字)

### 写作目标
介绍本系统采用的架构模式和设计理念,说明为什么选择这种架构。

### 内容要点

#### 前后端分离架构 (200字)
- **定义**:表现层与业务逻辑层的物理分离,通过HTTP API通信
- **优势**:
  - 前后端职责清晰,并行开发
  - 技术栈独立选型,灵活性强
  - 利于前后端分别优化与扩展
- **在本系统中的应用**:
  - React SPA负责用户交互与实时渲染
  - Node.js Express提供RESTful API
  - MongoDB Atlas作为云数据库服务

#### RESTful API设计理念 (200字)
- **核心原则**:资源导向、无状态、统一接口
- **本系统的API设计**:
  - GET /api/history - 查询历史记录
  - POST /api/ai/text-to-design - 文本生成设计稿
  - PUT /api/history/:id - 更新历史记录
- **优势**: 接口语义清晰,便于前后端联调与后期维护

#### 单页应用(SPA)架构特点 (200字)
- **定义**:整个应用加载单个HTML页面,通过JavaScript动态更新内容
- **技术实现**: React Router实现客户端路由,组件按需加载
- **用户体验优势**:
  - 页面切换无需刷新,响应迅速
  - 支持浏览器前进/后退
  - 离线缓存能力(Service Worker)

**写作要求**: 
- 以段落叙述为主,不要写成技术教程
- 重点在于"为什么选择"而非"如何使用"
- 适当提及替代方案并说明取舍理由

## 2.2 核心技术栈介绍 (约800字)

### 写作目标
系统介绍本项目使用的各项技术,突出技术选型的合理性。

### 内容框架

#### React 18框架特性 (250字)
- **函数式组件与Hooks**: 替代类组件,逻辑复用更优雅
  - useState/useEffect: 状态管理与副作用处理
  - useMemo/useCallback: 性能优化,避免不必要的重渲染
  - Custom Hooks: 封装可复用的业务逻辑
  
- **虚拟DOM(Virtual DOM)**: 
  - 声明式编程范式
  - Diff算法高效更新真实DOM
  - 在本系统中用于Design JSON的高效渲染

- **并发特性(Concurrent Features)**:
  - React 18引入的自动批处理、Suspense等
  - 提升大型应用的响应性能

- **TypeScript集成**:
  - 静态类型检查,减少运行时错误
  - Design JSON的类型定义(design-json.ts)确保数据结构正确性

**参考价值**: 可引用React官方文档或相关技术书籍

#### Node.js + Express后端技术 (200字)
- **Node.js特点**:
  - JavaScript全栈开发,前后端语言统一
  - 事件驱动、非阻塞I/O,适合高并发场景
  - npm生态丰富,模块化开发便捷
  
- **Express框架**:
  - 轻量级Web应用框架
  - 中间件机制(Middleware)便于功能扩展
  - 路由设计清晰,支持RESTful风格
  
- **在本系统中的角色**:
  - AI服务的代理与编排
  - 用户认证与会话管理
  - 文件上传处理与数据库操作

#### MongoDB文档数据库 (150字)
- **NoSQL特点**:
  - 灵活的Schema设计,适合Design JSON这种嵌套文档结构
  - 水平扩展能力强
  - MongoDB Atlas云服务,免去运维负担
  
- **为何选择MongoDB而非MySQL**:
  - Design JSON结构动态变化,关系型数据库Schema频繁变更代价高
  - 嵌套文档天然支持树形结构存储
  - 查询性能满足本系统需求

#### JWT认证机制 (100字)
- **JSON Web Token原理**:
  - 无状态认证,Token包含用户信息
  - 三部分组成:Header.Payload.Signature
  - 服务端无需存储Session,易于水平扩展
  
- **本系统的安全策略**:
  - 密码bcrypt加密存储
  - Token有效期7天,支持自动登录
  - 敏感操作二次验证

#### Zustand状态管理 (100字)
- **轻量级替代Redux**:
  - API简洁,Boilerplate代码少
  - 订阅机制精细,仅重渲染订阅的组件
  - 支持中间件(devtools、persist等)
  
- **在本系统中的应用**:
  - 全局Store维护currentDesignJson、historyList等核心状态
  - 会话分片存储,避免全局状态污染

**写作技巧**: 
- 每项技术按照"简介→特点→在本系统中的作用"三段式组织
- 用表格对比会更清晰(建议插入表2-1:技术选型对比表)
- 避免过度深入技术细节,保持概述性质

## 2.3 AI与大模型技术 (约600字)

### 写作目标
介绍AI与大模型的基础知识,特别是与本项目相关的技术背景。

### 内容要点

#### 大语言模型(LLM)基本原理 (150字)
- **Transformer架构**: Self-Attention机制捕捉长距离依赖
- **预训练+微调范式**: 大规模语料预训练,下游任务微调
- **涌现能力**: 模型规模扩大后展现出的推理、理解等能力
- **代表模型**: GPT系列、LLaMA、通义千问等

#### 多模态视觉模型能力 (150字)
- **视觉-语言模型(VLM)**: 同时理解图像与文本
- **图像理解能力**: 目标检测、OCR、布局分析
- **在本系统中的应用**: 
  - 解析上传的设计稿图片(PNG/JPG)
  - 提取布局结构、组件类型、样式属性
  - 映射为标准化的Design JSON

#### Prompt工程技术 (150字)
- **Prompt设计原则**:
  - 明确的角色设定(System Prompt)
  - 清晰的任务指令
  - 示例引导(Few-shot Learning)
  - 输出格式约束(JSON Schema)
  
- **本系统的Prompt工程**:
  - 四段式结构:系统指令+历史对话+当前状态+用户指令
  - Design JSON上下文注入策略
  - Chain-of-Thought引导增量修改

#### 通义千问模型简介 (100字)
- **阿里云大模型**: 中文理解能力优秀
- **qwen-turbo版本**: 8K上下文窗口,响应速度快
- **API调用方式**: HTTP RESTful接口,支持SSE流式输出
- **降级策略**: API异常时切换至本地伪数据,保障可用性

#### 算法微调背景 (50字)
- **为何需要微调**: 通用模型对Design JSON格式的理解不够精确
- **微调方法选择**: Prompt Engineering为主,辅以Few-shot示例
- **效果验证**: 增量修改准确率从初始60%提升至82%

**写作要求**: 
- 这一部分要为第五章的算法优化专题做铺垫
- 简述微调思路即可,详细过程留到第五章展开
- 引用相关文献说明大模型技术的发展脉络
