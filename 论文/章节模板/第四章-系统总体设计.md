## 4.1 技术选型说明 (约800字)

### 内容框架

#### 前端技术栈论证 (300字)
- **React 18 vs Vue 3 vs Angular**:
  - 选择React的理由:
    - 虚拟DOM机制适合Design JSON的高频更新场景
    - 丰富的生态系统(组件库、状态管理、开发工具)
    - TypeScript支持成熟,类型安全有保障
    - 团队熟悉度高,学习成本低
  
  - **Vite vs Webpack**:
    - Vite的极速冷启动(基于ESM)
    - HMR(热模块替换)响应速度快
    - 配置简洁,开箱即用

  - **Zustand vs Redux vs MobX**:
    - Zustand的轻量级(代码量少,API简洁)
    - 细粒度订阅控制,性能优秀
    - 内置中间件支持(devtools、persist)

#### 后端技术栈论证 (250字)
- **Node.js + Express**:
  - JavaScript全栈,前后端统一语言
  - 非阻塞I/O,适合AI服务的异步调用
  - 中间件机制灵活(auth、cors、error-handling)
  
- **MongoDB Atlas**:
  - Design JSON为嵌套文档结构,NoSQL天然适配
  - Schema灵活,便于迭代演进
  - 云服务免运维,自动备份与扩展

- **JWT认证**:
  - 无状态,适合前后端分离架构
  - Token自包含用户信息,减少数据库查询
  - 标准化实现,安全性有保障

#### AI服务选型论证 (150字)
- **通义千问 vs GPT-4 vs Claude**:
  - 通义千问的优势:
    - 中文理解能力强,适合国内用户
    - API价格亲民,适合毕设项目预算
    - 8K上下文窗口满足Design JSON注入需求
    - SSE流式输出支持良好
  
  - **降级策略考虑**:
    - 外部API依赖的风险(网络延迟、服务不可用)
    - 本地伪数据(mock)作为备选方案
    - 保证核心功能可用性

#### 技术栈总结 (100字)
用一段话概括:"综合考量开发效率、运行性能、生态成熟度与项目成本,本系统最终采用React 18+Vite+TypeScript构建前端,Node.js+Express+MongoDB搭建后端服务,通义千问大模型提供AI能力。这一技术组合在保证功能完整性的同时,兼顾了开发效率与系统可维护性..."

## 4.2 系统整体架构 (约1,500字) ⭐核心段落

### 写作目标
详细描述系统的分层架构、核心思想与数据流向,这是本章最重要的部分。

### 内容框架

#### 四层架构设计详述 (800字)

##### 第一层:表现层(Presentation Layer)
**职责**: 用户界面渲染与交互处理

**技术实现**:
- React 18单页应用(SPA),基于函数式组件与Hooks
- React Router v6实现客户端路由,功能模块按需加载
- CSS Modules + CSS Variables实现样式隔离与主题切换
- 三栏式布局:侧边栏(导航+历史)、对话区域(输入交互)、预览区域(编辑+预览)

**关键组件**:
- Layout.js: 整体布局容器,管理三栏宽度分配与响应式折叠
- Sidebar.js: 功能入口与历史记录列表
- ConversationArea.js: 对话消息列表与输入框
- PreviewArea.js: 设计预览/代码预览切换视图
- VisualEditor.js: 可视化编辑器核心控制器

**设计原则**:
- 表现层仅负责UI渲染与事件分发,不包含业务逻辑
- 通过Custom Hooks封装可复用的交互逻辑(useSelection、useDragAndDrop等)
- 使用Context API传递全局状态(designJson、selectedId)

##### 第二层:业务逻辑层(Business Logic Layer)
**职责**: Design JSON的处理、验证、转换与状态管理

**核心模块**:

1. **设计JSON处理器**:
   - designJsonUtils.js: JSON树的遍历、查找、更新、删除操作
   - designJsonAdapter.js: 不同版本格式的兼容转换
   - styleConverter.js: Design JSON样式属性→CSS字符串的转换
   
2. **状态管理Store**(store.js):
   ```javascript
   // 核心状态结构
   {
     currentDesignJson: DesignJSON | null,  // 当前设计稿
     historyList: HistoryRecord[],           // 历史记录列表
     selectedId: string | null,              // 当前选中组件ID
     conversations: Message[],               // 当前对话消息
     user: User | null,                      // 当前登录用户
     sessionId: string                       // 当前会话ID
   }
   ```
   
   - Zustand Store按会话分片(sessions[sessionId])
   - 支持持久化(persist中间件保存至localStorage)
   - DevTools集成,便于调试

3. **自定义Hooks**:
   - useHistory(): 历史记录的增删改查
   - useSelection(): 组件选中与高亮逻辑
   - useDragAndDrop(): 拖拽操作的状态机
   - useKeyboardShortcuts(): 快捷键绑定(Ctrl+Z/Y/Delete)

**设计原则**:
- 业务逻辑抽象为纯函数,便于测试与复用
- 通过Hooks实现逻辑复用,避免组件间 prop drilling
- 状态变更通过统一接口(dispatch),确保可追踪性

##### 第三层:数据层(Data Layer)
**职责**: 数据的持久化存储与运行时缓存

**混合存储策略**:

1. **运行时数据**(前端内存):
   - 当前Design JSON驻留于Zustand Store
   - 编辑历史栈(undo/redo)使用不可变数组
   - 优势:实时响应,无网络延迟
   
2. **持久化数据**(MongoDB Atlas):
   - 用户信息(users集合)
   - 历史记录(histories集合):嵌套存储完整Design JSON
   - 上传文件(images/avatars):GridFS或Base64编码
   
3. **本地缓存**(localStorage):
   - JWT Token
   - 用户偏好设置(主题模式、侧边栏展开状态)
   - 草稿数据(未保存的设计稿)

**数据一致性保障**:
- 自动保存机制:防抖(2秒)+强制保存(页面关闭前)
- 乐观更新:先更新UI再同步后端,失败时回滚
- 冲突检测:基于时间戳的最后写入胜出策略

##### 第四层:AI服务层(AI Service Layer)
**职责**: AI能力的封装与编排

**架构设计**:
- **统一服务接口**: ai.js路由定义RESTful端点
- **适配器模式**: qwenService.js封装通义千问API调用细节
- **降级策略**: mockDesignGenerator.js提供本地伪数据

**核心能力**:
1. 文本生成(text-to-design): 自然语言→Design JSON
2. 图片生成(image-to-design): 图片解析→Design JSON
3. 代码生成(design-to-code): Design JSON→前端代码

**Prompt工程中心**: conversationManager.js
- 构建四段式Prompt(系统指令+历史+当前状态+用户指令)
- Design JSON上下文注入
- 对话历史管理(保留最近3轮)

**设计原则**:
- AI服务通过HTTP API与业务层解耦
- 统一错误处理与重试机制(指数退避)
- 流式响应(SSE)提升用户体验

#### 核心架构思想 (400字)

##### Single Source of Truth (单一真实数据源)
**定义**: Design JSON是系统中页面设计的唯一权威表示,所有功能模块均围绕该数据模型展开操作。

**实现方式**:
- 全局Store维护唯一的currentDesignJson对象
- AI生成、用户编辑、历史恢复都修改同一份数据
- 预览区域从Store读取数据并渲染
- 属性面板编辑直接更新Store中的节点属性

**优势**:
- 消除数据不一致风险(不存在多份副本同步问题)
- 状态变更可追溯(每次修改都有明确来源)
- 简化架构复杂度(无需复杂的同步机制)

**类比**: 类似Redux的"Single Source of Truth"理念,但应用于设计数据而非应用状态。

##### 数据驱动闭环 (Data-Driven Closed Loop)
**工作流程**:
```
用户输入(文本/图片) 
    → AI生成(Design JSON)
    → 可视化预览(渲染引擎)
    → 用户编辑(属性面板/拖拽)
    → 更新Design JSON
    → 实时重新渲染
    → (可选)生成代码(Design JSON→React/Vue/HTML)
```

**关键特性**:
- **双向绑定**: 视图↔数据的实时同步
- **操作归一化**: 所有异构操作(生成/编辑/恢复)统一为数据变换
- **版本可控**: 每次操作产生新的不可变快照,支持撤销/回溯

**对比传统模式**:
- 传统:Figma设计稿→开发者手写代码→浏览器预览→发现问题→返回Figma修改
- 本系统:AI生成/图片上传→可视化编辑(所见即所得)→一键导出代码

##### 高内聚低耦合 (High Cohesion, Low Coupling)
**内聚性表现**:
- VisualEditor专注于编辑交互逻辑
- DesignRenderer专注于渲染逻辑
- qwenService专注于AI调用封装

**耦合度控制**:
- 层间通过标准化接口通信(HTTP API、Function Call)
- 模块间通过依赖注入解耦(如AI服务可通过配置切换)
- 使用Event Emitter或Callback处理跨模块通知

#### 4.3.1 数据层/存储模块架构 (500字)

##### MongoDB Schema设计

**users集合** (用户表):
```javascript
{
  _id: ObjectId,              // 主键
  email: String,              // 邮箱(唯一索引)
  password: String,           // bcrypt加密哈希
  username: String,           // 用户名
  avatar: String,             // 头像URL
  isVerified: Boolean,        // 邮箱是否验证
  verificationToken: String,  // 验证Token
  createdAt: Date,
  updatedAt: Date
}
```

**histories集合** (历史记录表):
```javascript
{
  _id: ObjectId,
  userId: ObjectId,           // 关联用户(索引)
  sessionId: String,          // 会话ID(索引,唯一)
  type: String,               // 'text-to-design' | 'image-to-design' | 'design-to-code'
  
  input: {
    text: String,             // 用户输入的文本
    imageUrl: String          // 上传图片的URL
  },
  
  designJson: {               // 完整的Design JSON(嵌套文档)
    version: "1.0",
    type: "design-json",
    metadata: {...},
    root: ComponentNode       // 根节点(递归嵌套children)
  },
  
  codeResult: {               // 生成的代码结果(可选)
    framework: 'react' | 'vue' | 'html',
    files: [
      {
        path: "src/App.jsx",
        content: "...",
        language: "javascript"
      }
    ]
  },
  
  conversations: [            // 对话记录(可选)
    {
      role: 'user' | 'assistant',
      content: String,
      timestamp: Date,
      designJson: Object      // 版本锚点
    }
  ],
  
  createdAt: Date,
  updatedAt: Date
}
```

**Schema设计理由**:
- 嵌套文档存储Design JSON:避免关联查询,读取效率高
- sessionId唯一索引:确保会话隔离,避免数据冲突
- userId索引:快速查询某用户的所有历史
- conversations数组:保存对话上下文,支持恢复

##### 存储策略详解

**运行时状态**(前端):
```
Zustand Store (内存)
├── sessions[sessionId]
│   ├── designJson: DesignJSON     // 当前设计稿(热数据)
│   ├── conversations: Message[]   // 对话消息
│   ├── selection: SelectionState  // 选中状态
│   └── isModified: boolean        // 是否有未保存修改
```

**持久化数据**(MongoDB):
- 冷数据:历史记录、用户信息
- 写入时机:自动保存(防抖2s)、手动保存、页面关闭
- 读取时机:登录加载历史列表、点击记录恢复

**本地缓存**(localStorage):
- JWT Token(7天有效期)
- 用户偏好(主题、布局)
- 未保存草稿(防止意外关闭丢失)

##### 数据一致性机制
- **写操作顺序**: 先更新UI(乐观更新) → 异步保存至MongoDB → 失败则回滚并提示
- **并发控制**: 同一会话串行化处理,不同会话独立隔离
- **冲突解决**: 最后写入胜出(Last Write Wins),基于updatedAt时间戳

#### 4.3.2 AI生成模块/文件管理服务 (700字)

##### 文本生成服务 (text-to-design)
**接口定义**:
```
POST /api/ai/text-to-design
Request Body: {
  prompt: string,            // 用户输入的文本
  sessionId: string,         // 会话ID
  currentDesignJson: object  // 当前的Design JSON(可选,用于增量修改)
}

Response: {
  success: boolean,
  designJson: object,        // 生成的Design JSON
  replyText: string,         // AI回复文本(用于显示在对话框)
  usage: {                   // Token消耗统计
    promptTokens: number,
    completionTokens: number,
    totalTokens: number
  }
}
```

**处理流程**:
1. 接收用户输入与当前Design JSON(如有)
2. 调用conversationManager.buildMessages()构建Prompt
3. 注入四段式Prompt(系统指令+历史对话+当前状态+用户指令)
4. 调用qwenService.callQwenAPI()发送请求
5. 采用SSE(Server-Sent Events)流式接收响应
6. 解析AI返回的JSON字符串,校验格式合规性
7. 返回标准化的Design JSON对象

**质量保障**:
- Prompt模板固化,确保输出格式一致
- JSON Schema验证,拒绝非法格式
- 超时控制(30秒),避免长时间等待
- 错误降级:API异常时返回mock数据

##### 图片生成服务 (image-to-design)
**接口定义**:
```
POST /api/ai/image-to-design
Request Body (multipart/form-data):
  image: File,                // PNG/JPG文件(≤5MB)
  sessionId: string

Response: {
  success: boolean,
  designJson: object,
  imageUrl: string,           // 保存后的图片URL
  replyText: string
}
```

**处理流程**:
1. 接收上传的图片文件
2. 校验格式(PNG/JPG)与大小(≤5MB)
3. 保存至服务器uploads/images/目录(或云存储OSS)
4. 将图片转为Base64编码(或使用URL引用)
5. 构建视觉理解Prompt("请分析这张UI截图的结构...")
6. 调用通义千问VL(视觉语言模型)
7. 模型提取布局结构、识别组件类型、推断样式属性
8. 映射为标准Design JSON格式
9. 返回结果并在预览区渲染

**技术挑战与应对**:
- **挑战1**: 复杂排版识别准确率低
  - 应对:Prompt中引导模型优先识别主要区域,忽略细节装饰
  
- **挑战2**: 样式属性推断不准确(颜色、字号)
  - 应对:Post-processing规则修正(如主色调取Top-3高频色)

##### 代码生成服务 (design-to-code)
**接口定义**:
```
POST /api/ai/design-to-code
Request Body: {
  designJson: object,         // 当前确认的Design JSON
  framework: 'react' | 'vue' | 'html'
}

Response: {
  success: boolean,
  codeResult: {
    framework: string,
    files: [
      {
        name: "App.jsx",
        content: "...",
        language: "javascript"
      },
      {
        name: "App.css",
        content: "...",
        language: "css"
      }
    ]
  }
}
```

**处理流程**:
1. 接收Design JSON与目标框架参数
2. 根据框架选择对应的代码模板
3. 递归遍历Design JSON树,逐节点生成对应代码
4. 样式属性转换为CSS(内联style或CSS类名)
5. 组件映射:container→div, text→span/p, button→<Button>, input→<Input>
6. 生成完整的文件结构(App.jsx + App.css + index.html等)
7. 返回文件列表供前端预览或下载

**模板引擎设计**:
- React模板: 函数式组件 + Hooks + JSX语法
- Vue模板: SFC(单文件组件) + <template>语法
- HTML模板: 原生HTML + CSS + Vanilla JS

##### AI服务编排与降级策略
**统一错误处理**:

- 根据用户输入关键词匹配最相似模板
- 支持随机变化(颜色、间距)增加多样性

#### 4.3.3 可视化编辑模块 (600字)

##### DesignRenderer渲染引擎
**核心职责**: 将Design JSON树形结构递归映射为React组件树

**算法伪代码**:
```
function renderNode(node, path):
  1. 根据 node.type 查找对应的 React 组件
     component = componentMap[node.type] || DefaultDiv
     
  2. 转换样式属性
     cssStyle = convertStyleToCSS(node.style)
     // 示例: {padding: [10,20,10,20]} → "padding: 10px 20px"
     
  3. 判断是否被选中
     isSelected = (node.id === selectedId)
     
  4. 渲染当前节点
     return <Component
       key={node.id}
       style={cssStyle}
       className={isSelected ? 'selected' : ''}
       onClick={() => handleSelect(node)}
     >
       {node.children?.map((child, index) =>
         renderNode(child, [...path, index])
       )}
     </Component>
```

**性能优化策略**:
- **React.memo**: 包裹DesignNode组件,自定义比较函数

```javascript
// 外部设计稿更新时同步
useEffect(() => {
  if (initialDesignJson !== designJson) {
    setDesignJson(initialDesignJson);
    // 验证选中节点是否仍存在
    const exists = findNodeById(initialDesignJson, selectedIdRef.current);
    if (!exists) {
      selectedIdRef.current = initialDesignJson.root?.children[0]?.id;
    }
  }
}, [initialDesignJson]);
```
