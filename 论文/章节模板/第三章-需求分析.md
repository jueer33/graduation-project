# 第三章 需求分析 - 写作指南

## 章节概述
本章从用户角度出发,系统地分析系统应该具备的功能需求和非功能需求。通过用例分析、场景描述等方式,明确系统的设计目标和验收标准。

**总字数**: 约2,500字 (占全文10%)
**重要程度**: ⭐⭐⭐⭐ (承上启下,连接背景与设计)

## 3.1 功能需求分析 (约1,800字)

### 写作目标
全面梳理系统的功能需求,使用用例图、用户故事等方法进行结构化表达。

### 内容框架

#### 3.1.1 设计稿生成需求 (700字)

##### 文本生成设计稿
**用户故事**: 作为一名前端开发者,我希望通过自然语言描述页面需求,系统能自动生成可视化的设计稿,以便快速验证设计想法。

**功能描述**:
- 用户在对话区域输入自然语言描述(如"创建一个登录页面,包含用户名、密码输入框和登录按钮")
- 系统调用AI服务,将文本转换为结构化的Design JSON
- 生成的Design JSON实时渲染在右侧预览区域
- 支持多轮对话,可在已有基础上提出修改意见

**输入输出示例**:
- 输入: "创建一个电商商品列表页,左侧筛选栏,右侧商品卡片网格"
- 输出: 符合Design JSON规范的完整JSON对象(见json结构.md中的示例)

**质量要求**:
- 响应时间<3秒
- Design JSON格式合规率100%
- 生成结果与用户描述的一致性>80%

##### 图片生成设计稿
**用户故事**: 作为一名UI设计师,我希望上传设计稿图片(PNG/JPG),系统能自动解析并转换为可编辑的设计稿,以便在代码实现前进行调整。

**功能描述**:
- 支持拖拽上传或点击选择文件
- 文件格式限制:PNG/JPG,最大5MB
- 上传后显示缩略图预览
- 调用多模态视觉模型解析图片
- 提取布局结构、组件类型、样式属性
- 生成对应的Design JSON并在预览区渲染

**技术挑战**:
- 图片理解的准确性(布局识别、组件边界检测)
- 视觉样式到CSS属性的映射
- 复杂排版(网格、Flex)的结构还原

**用例场景**:
- 场景1: 设计师在Figma完成高保真原型,导出PNG上传,系统自动生成可编辑版本
- 场景2: 用户看到喜欢的网页截图,上传后快速生成类似布局的设计稿

##### 输入预处理与校验
- 文本输入:长度限制、敏感词过滤、空值检查
- 图片输入:格式校验、大小限制、Base64编码
- 异常提示:友好的错误信息,引导用户修正

#### 3.1.2 可视化编辑需求 (600字)

##### 实时预览渲染
**功能描述**:
- DesignRenderer递归遍历Design JSON树形结构
- 动态映射为React组件树
- 支持容器(container)、文本(text)、按钮(button)、输入框(input)、图片(image)、卡片(card)、分割线(divider)七种组件类型
- 完整覆盖Flex布局体系(flexDirection、justifyContent、alignItems、gap等)
- CSS样式属性实时生效(颜色、间距、圆角、阴影等)

**渲染质量要求**:
- 组件视觉效果与真实网页一致度>95%
- 复杂设计稿(100+节点)渲染延迟<100ms
- 样式更新后重渲染延迟<80ms

##### 组件交互操作
**功能清单**:

1. **组件选中与高亮**
   - 点击组件显示蓝色边框高亮
   - 右侧属性面板同步显示该组件的所有属性
   - 支持键盘快捷键(Tab切换、Delete删除)

2. **属性实时编辑**
   - 五类属性分组:基础(宽高、背景色)、布局(display、flex)、间距(padding、margin、gap)、外观(圆角、阴影、边框)、文本(字体大小、颜色、粗细)
   - 修改即时反馈至预览区(所见即所得)
   - 数值输入框、颜色拾取器、下拉选择器等多种控件

3. **拖拽调整层级**
   - 拖拽组件改变父子关系或兄弟顺序
   - 智能区域判定:上25%-前、中50%-内部、下25%-后
   - 类型约束:仅容器类型(page/container/card)可接收子组件
   - 视觉反馈:拖拽时显示插入指示线和高亮区域

4. **撤销/重做机制**
   - 支持20步操作历史深度
   - Ctrl+Z撤销、Ctrl+Y重做
   - 基于不可变数据结构的快照存储

5. **文本内容编辑**
   - 双击文本组件进入编辑模式
   - AI生成的内容(isGenerated=true)锁定编辑权限
   - 手动添加的组件可自由编辑

**交互体验指标**:
- 选中响应时间<50ms
- 属性编辑反馈延迟<16ms(1帧)
- 拖拽帧率≥55fps
- 误操作率<10%

#### 3.1.3 代码生成与导出需求 (500字)

##### 多框架代码生成
**功能描述**:
- 基于确认后的Design JSON,生成指定前端框架的目标代码
- 支持三种框架:React(Vite)、Vue(Vite)、原生HTML(CSS+JS)
- 代码结构符合各框架的最佳实践
- 包含必要的注释,提高可读性

**代码质量要求**:
- 语法正确性100%
- 组件拆分合理(单文件组件)
- 样式与逻辑分离
- 响应式布局支持

##### 代码预览与运行
**功能描述**:
- 右侧区域切换至代码视图
- 左侧文件树展示生成的文件结构(App.jsx、App.css等)
- 点击文件节点切换显示对应代码
- 语法高亮显示(使用第三方库如Prism.js/highlight.js)
- iframe内实时运行预览,查看实际效果

##### 项目导出
**功能描述**:
- 一键下载完整项目ZIP包
- 包含package.json、源码、配置文件
- 解压后可直接npm install && npm run dev运行
- (可选)支持在线部署至Vercel/Netlify

**用例场景**:
- 场景1: 用户通过文本生成登录页设计稿 → 可视化调整配色和间距 → 一键导出React项目
- 场景2: UI设计师上传原型图 → 解析为可编辑设计稿 → 开发者调整细节 → 导出Vue代码

#### 3.1.4 用户管理与历史记录 (300字)

##### 用户系统
- 注册:邮箱注册,密码强度验证(至少8位,包含字母数字)
- 登录:邮箱+密码,返回JWT Token
- Token鉴权:所有敏感接口携带Authorization头
- 自动登录:Token有效期内免重复登录
- 用户信息:头像上传、昵称修改

##### 历史记录管理
- 自动保存触发时机:
  - 新对话生成完成后
  - 功能模块切换前
  - 页面关闭/刷新时(beforeunload事件)
  
- 历史列表展示:
  - 按时间倒序排列
  - 显示标题摘要、创建时间、类型标签(文本/图片/代码)
  - 支持删除单条记录
  
- 会话恢复:
  - 点击历史记录加载对应Design JSON
  - 恢复对话上下文
  - 可继续编辑或生成代码

- 数据隔离:
  - 基于userId关联,不同用户数据隔离
  - 基于sessionId会话分片,避免状态污染

## 3.2 非功能需求分析 (约500字)

### 写作目标
从性能、易用性、安全性等维度明确系统的质量属性要求。

### 内容框架

#### 性能需求 (200字)
- **响应时间**:
  - AI文本生成:<3秒(首字节时间<1秒,流式输出)
  - AI图片生成:<5秒
  - 可视化编辑操作:<100ms
  - 页面加载时间(FCP):<2秒
  
- **并发能力**:
  - 支持50个在线用户同时使用
  - AI服务队列管理,避免过载
  
- **资源占用**:
  - 内存占用<200MB(前端)
  - Design JSON大小限制:<1MB(约500个组件节点)

#### 易用性需求 (150字)
- **学习成本**:
  - 新用户5分钟内掌握基本操作
  - 提供新手引导(Tooltip、Skeleton Screen)
  
- **操作流畅度**:
  - 动画过渡流畅(60fps)
  - 加载状态反馈(Skeleton屏、Loading动画)
  - 错误提示友好(Toast通知,非alert弹窗)
  
- **界面设计**:
  - 支持亮色/暗色主题切换
  - 响应式布局(≥1200px三栏、992px折叠侧边栏、<768px单栏)
  - 符合现代UI设计规范(Ant Design风格)

#### 可扩展性需求 (100字)
- **组件扩展**:
  - 新增组件类型仅需修改componentMap映射表
  - Design JSON Schema支持自定义字段扩展
  
- **AI模型替换**:
  - 适配器模式封装AI服务调用
  - 切换模型只需修改配置文件
  
- **框架支持扩展**:
  - 代码生成器采用模板引擎
  - 新增框架支持只需编写模板文件

#### 安全性需求 (50字)
- 用户密码bcrypt加密存储
- JWT Token签名防篡改
- 文件上传类型与大小限制
- API请求频率限制(Rate Limiting)
- XSS攻击防护(React内置转义)

## 写作注意事项

1. **用户视角**:始终从"用户需要什么"角度描述,而非"系统有什么功能"
2. **场景化**:多用具体的用户故事和使用场景,避免抽象的功能罗列
3. **可度量**:尽量量化需求(响应时间<3秒、准确率>80%),便于后续测试验证
4. **完整性**:覆盖所有已实现的功能,不遗漏
5. **合理性**:需求要符合技术可行性和时间成本
